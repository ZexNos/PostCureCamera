    ## 1. UI Style Guide

    (source: docs/postcure_camera_UI_style_guide.md)


# Post‑Cure Camera — UI Style Guide (v2025‑11‑01)

**Target display:** ST7789, landscape **160×128 px**, dark theme.  
**Global goals:** читаемость с 50–100 см, минимализм, стабильная сетка, единая палитра.

---

## 1) Палитра (RGB)

- **Фон окна:** `35,35,35`  
- **Текст основной:** `220,220,220`  
- **Статус‑бар (верх):** `46,46,46`  
- **Серый вторичный:** `125,125,125`

**Акценты:**
- **Оранжевый UI (accent):** `255,140,0`
- **Температура (чуть краснее):** `255,120,90`
- **UV (фиолетовый):** `180,90,255`
- **MQTT online индикатор:** `0,200,0`

**Аварии / подсистемы:**
- **Общий WARNING (красный):** `255,85,85`
- **Heater cause block:** `255,112,67`
- **UV cause block:** `217,128,255`
- **Fan cause block:** `111,207,255`

---

## 2) Верхняя статус-строка

- Высота: 16 px (`y = 0..15`), ширина = 160 px.
- Фон: цвет статус-бара, разделительная линия снизу.
- Левая зона (примерно `x = 4..40`):
  - иконка Wi-Fi (off: серые дуги с крестом; on: залитые зелёные «палочки»),
  - иконка MQTT в виде точки (off: контур, on: заливка фиолетовым).
- Центр: заголовок экрана (`RUN`, `OPTIONS`, `TEST`, `AUTOTEST`, `UV`, `DRY_...`), выровнен по центру строки.
- Правая зона:
  - HEAT — оранжевая точка (on) / контур (off),
  - UV — фиолетовая точка (on) / контур (off).
- Текст и иконки — только в один пиксельный грид, без смещения между кадрами.


## 3) Типографика

- Базовый шрифт: **DejaVu Sans** (или системный fallback).  
- Размеры: `S=8`, `M=10`, `L=12`, `XL=14` pt.  
- В одном экране не смешивать >2 размеров, кроме заголовка.

---

## 4) Сетка и отступы

- Внутренние поля экранов: **6–8 px**.  
- Строка состояния сверху: **14 px** высоты.  
- Нижняя навигация: **14 px** высоты, текст `"↑ ↓   Click = ..."` по центру.  
- Кнопки действий: высота **16–18 px**, радиус скругления **3–6 px**.
- Прогресс‑бар: ширина **120 px**, высота **6–8 px**.

---

## 5) Обязательная верхняя строка (Top Bar)

Слева: значок Wi‑Fi (дуги), рядом статус **MQTT** (зелёная точка или контур).  
Справа: маленькие иконки **🔥** (heater) и **🌕** (UV) — только цвет включения  
(оранжевый и фиолетовый соответственно), выключено — серым.

---

## 6) Цветовая логика

- **Оранжевый** — нейтральный акцент UI, прогресс‑полосы, курсор выбора.  
- **Температура** — оранжево‑красный (`255,120,90`).  
- **UV** — фиолетовый (`180,90,255`).  
- **Аварии** — заголовок и кнопка **красные** (`255,85,85`),  
  **причина** — плашка цвета подсистемы (heater/uv/fan).

---

## 7) Компоненты

- **ListItem**: прямоугольник `48–55` тёмный, подсвеченный — обводка акцентом; слева — название, справа — параметры (`S`‑шрифтом).  
- **Tile** (PRESETS root): два квадрата **64×44 px**, зазор **14 px**.  
- **Buttons**: вторичная — тёмно‑серая, основная — оранжевая (или красная для ошибок), текст контрастный.  
- **Progress**: тонкая рамка, ровный fill по цвету состояния.

---

## 8) Экраны (ключевые правила)

- **RUN VIEW**: крупные **UV% (фиолетовый)** и **T air (оранжево-красный)** на одной строке; ниже — прогресс.  
- **OPTIONS**: список (`Pause/Resume`, `Stop`, `Adjust`, `Back to Progress`).  
- **ERROR v4**: красный заголовок + красная кнопка `OK`; плашка‑причина подсистемным цветом; без крестика в заголовке.  
- **CONNECTION STATUS**: `Connecting` (голубой), `Connected` (салатовый), `✖ FAILED` (красный, объяснение в две строки).  
- **OTA VIEW**: `Downloading / Installing / Success / Failed` с прогресс‑баром.

---

## 9) Консистентность

- Верхняя статус-строка **не меняется** между экранами (расположение иконок и заголовка фиксировано).  
- Везде **альбомная** ориентация `160×128`.  
- Минимум текста, максимум читабельности.

---

## 10) Файлы рендерера

Используйте модуль `postcure_ui_renderer.py` (см. рядом), чтобы получить идентичную отрисовку во всех ветках.

    ## 2. UI Lists & Diff Rendering — Правила обновления

    (source: docs/README_Files/UI_Lists_and_Diff_Render.md)

    # UI Lists & Diff Rendering — Правила обновления (v1.0)
**Updated:** 2025-11-09 11:04

Цель: единый контракт для экранов и предсказуемая, безмерцательная отрисовка списков/контента.

---

## Геометрия экрана (ST7789 160×128, landscape)
- TOP-bar: `y = 0..15` (фиксированная полоска статусов + заголовок).  
- CONTENT: `y = 16..127` (всё, что рисуют экраны).

-TOP-bar рисуется драйвером Display **по delta** (только при изменении статусов/заголовка). CONTENT-область отвечает за diff-отрисовку списков/контента.

---

## Контракт экрана
Каждый экран реализует методы:
- `begin(disp, enc)` — привязка зависимостей, сброс состояния.
- `drawStatic()` — фон/рамки/неподвижные элементы, 1 раз.
- `drawDynamic()` — отрисовать **только изменившиеся** элементы.
- `onEvent(ev)` — обработка вращений/кликов.
- `tick()` — таймеры/анимация без `delay()`.

Экраны **не** трогают TOP/BOTTOM напрямую — только CONTENT.

---

## Списки: `ListCtx` (понятия)
Рекомендуемый набор полей:
- `y0` — верхняя координата списка в CONTENT.
- `vis` — число видимых строк.
- `total` — всего элементов.
- `first` — индекс верхнего элемента.
- `cursor` — текущий выбранный элемент (0..total‑1).
- `labels[]` — массив текстов.
- `prevFirst`, `prevCursor` — кеш для diff‑отрисовки.

Навигация энкодером меняет `cursor` и, при необходимости, `first`.

---

## Diff‑отрисовка списка (правила)
- Если `first` или `cursor` не изменились → ничего не перерисовывать.
- Если изменилось — перерисовать **только** ранее подсвеченную строку и новую активную, а также строки, вошедшие/вышедшие из окна при прокрутке.
- Полный `fillScreen()` запрещён. CONTENT очищается и обновляется **частично**.

---

## Рекомендации по шрифтам/иконкам
- Использовать монотонные размеры и сетку пикселей из style‑гайда — текст не должен «дрожать» при перерисовке.
- Цвета — только из `Colors.h`.

---

## Диагностика (DBG_RENDER)
Счётчики для freeze‑тестов:
- `full_clear_count` — должен быть **0** на навигации по спискам.
- `diff_draw_calls` растёт при прокрутке.
- `topbar_redraw_count` растёт **только** при изменении статусов/заголовка.

    ## 3. UI Feature Sandbox & Modular Testing Methodology

    (source: docs/README_Files/UI_Feature_Sandbox_Methodology.md)

    # Post‑Cure Camera — UI Feature Sandbox & Modular Testing Methodology
**Updated:** 2025-11-08 16:46

---

## 🎯 Цель
Обеспечить изолированное тестирование **каждого UI‑экрана или функции** без влияния остальных модулей.  
Экран или фича считается полностью отлаженной, если она проходит песочницу и может быть подключена к UiRouter без изменений драйверов.

---

## 1️⃣ Базовые условия
- Экран ST7789 и энкодер EncButton **заморожены** и работают стабильно.
- Драйверы DisplayST7789 / EncoderDrv / Colors.h / Pins.h не изменяются.
- Песочница запускается в режиме `SCREEN_SANDBOX` (см. ниже).

---

## 2️⃣ Универсальный интерфейс экрана
Каждый экран реализует общий контракт `IScreen`:

```cpp
struct IScreen {
  virtual void begin(DisplayST7789* d, EncoderDrv* e) = 0;
  virtual void drawStatic() = 0;      // фон/рамки, один раз
  virtual void drawDynamic() = 0;     // только изменившиеся элементы
  virtual void onEvent(EncEvent ev) = 0;
  virtual void tick() = 0;            // таймеры/анимации без delay()
  virtual ~IScreen() {}
};
```
Все экраны рисуют **только CONTENT‑зону** (y=14..113). Верхняя/нижняя полосы принадлежат Display.

---

## 3️⃣ Песочница (Screen Harness)
Используется для теста отдельного экрана.

```cpp
#define SCREEN_SANDBOX 1
#define SCREEN_UNDER_TEST  WiFiSetupScreen   // ← меняем для каждой фичи

#if SCREEN_SANDBOX
static SCREEN_UNDER_TEST scr;
void App::begin(){ display.begin(); encoder.begin(); scr.begin(&display,&encoder); scr.drawStatic(); }
void App::tick(){ encoder.tick(); scr.tick(); scr.drawDynamic();
  if (auto ev = encoder.pollEvent(); ev!=EncEvent::None) scr.onEvent(ev);
}
#endif
```
Меняешь `SCREEN_UNDER_TEST` → можно тестить любой экран из `ui/screens/*` без правок роутера.

---

## 4️⃣ Инъекция состояния (моки)

Для проверки отображения без сервисов:
```cpp
AppState mock{ .wifiConnected=true, .mqttConnected=false, .heaterOn=false, .uvOn=false };
display.drawTopBar(mock);              // кэш сам решит, перерисовывать ли
```

---

## 5️⃣ Политика отрисовки
- `drawStatic()` — один раз при создании/обновлении макета.
- `drawDynamic()` — только изменившиеся данные; никаких `fillScreen()`.
- Иконки статусов — обновляются только по delta (через кэш Display).

---

## 6️⃣ Definition of Done (для экрана/фичи)
- ✅ Реализованы все методы интерфейса.
- ✅ CONTENT‑only; без прямых вызовов top/bottom‑bar.
- ✅ Нет полного клира; дифф‑отрисовка работает.
- ✅ Навигация через энкодер корректна.
- ✅ Цвета/шрифты из Colors.h и style‑гайда.
- ✅ Песочница прогнана (вращение, клики, стресс ≥60 с).

---

## 7️⃣ Интеграция в систему
После успешного теста в песочнице:
- Добавить маршрут в `UiRouter`:
  ```cpp
  case Route::WiFiSetup: current = &wifiSetupScreen; break;
  ```
- Добавить при необходимости переходы в `App`.
- Выключить песочницу (`#undef SCREEN_SANDBOX`).
- Система автоматически подхватит экран в основном цикле.

---

## 8️⃣ Преимущества методики
- Можно разрабатывать и отлаживать любой экран независимо от MQTT, Wi‑Fi и датчиков.  
- Гарантируется единообразие интерфейса.  
- После успешного теста интеграция в UiRouter/App — чисто декларативная.  
- Позволяет делать **регрессионные тесты экранов** (песочница как unit‑runner).

---

## 📘 Рекомендуемая структура
```
ui/
  screens/
    RunView.cpp
    Options.cpp
    WiFiSetup.cpp
    MqttSettings.cpp
    ...
  sandbox/
    ScreenSandbox.h/.cpp     ← шаблон песочницы
```

---

## 9️⃣ Чек‑лист перед «заморозкой» экрана
- [ ] drawStatic()/drawDynamic()/onEvent()/tick() реализованы.
- [ ] Все цвета/шрифты из Colors.h.
- [ ] Навигация и события соответствуют ТЗ.
- [ ] Прошёл тест песочницы без ошибок 60 секунд.
- [ ] Визуально нет артефактов, flicker и дублирующих draw.
- [ ] Экран интегрирован в UiRouter и App без конфликтов.

---

**После выполнения чек‑листа экран считается готовым к заморозке и не требует изменений при дальнейшем росте проекта.**
